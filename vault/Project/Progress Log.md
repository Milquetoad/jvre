## 2026-06-16 тАФ Sampler config: per-image filter (NEAREST/LINEAR) ЁЯФНтЬЕ
- **The pull from the image-loading beat, delivered:** a `Filter` enum (NEAREST/LINEAR, bare-enum style like `Cull`) chooses a texture's sampler filter. Threaded through `Texture.create`/`load` and `Renderer.createImage`/`loadImage` (each with a no-filter overload). **Defaults by intent**: `createImage` (you authored the pixels) -> NEAREST (crisp -- keeps the cube checker + generated images sharp, no regression); `loadImage` (a decoded asset) -> LINEAR (smooth scaling). Override either explicitly. Verified on the 4090.
- **Honest scope -- what filtering is NOT:** it does not invent detail. LINEAR vs NEAREST only changes the *artifact character* at texel boundaries (smooth blur vs hard blocks); flat regions are identical either way. A 64px source enlarged looks low-res regardless -- that needs a higher-res asset, not a filter (no super-resolution). So the demo's flat 64px badge barely changes between filters; the feature is still correct + standard, just not showcased by that asset. Decision: ship as-is (didn't swap the demo asset). Lesson added to [[Textures - Images, Views and Samplers]].
- **Still future (the rest of "sampler config"):** mipmaps + anisotropy (improve *minification*, the downscaling case) -- a later beat; this knob covers magnification/upscaling.

## 2026-06-15 тАФ Phase 3a: SDF curves -- ellipses + rings go analytic ЁЯЯбтЬЕ
- **The jaggies are gone.** `fillEllipse`/`strokeCircle`/`strokeEllipse` moved off tessellation onto the SDF path -- crisp at any size, under any transform, **independent of MSAA** (the user spotted them aliasing at MSAA 1). Each is now **ONE 6-vertex bounding quad** instead of a radius-scaled fan/ring.
- **The shader (`shape2d.frag`)** gained two helpers: `ellipseSDF` (Inigo Quilez's gradient-corrected approximate ellipse distance -- EXACT for circles, accurate enough for a 1px AA ramp; there is no closed-form ellipse distance) and `coverageFromSdf` (the shared ~1-screen-pixel ramp using `length(dFdx, dFdy)`, which mode 1 now reuses too). Two new modes: **4** = ellipse FILL, **5** = ellipse RING (`abs(d) - halfThickness`, an annulus around the centerline; the half-thickness rides in the corner-radius slot).
- **`Renderer2D`**: generalized `sdfBox` -> `sdfQuad(cx, cy, padX, padY, halfX, halfY, cornerRadius, mode, c)` (every SDF mode funnels through it now); `fillEllipse` emits mode 4 (half = radii, pad = radius + 1.5px), `strokeEllipse` emits mode 5 (half = centerline radii, cornerRadius = half-thickness, pad clears the outer rim + 1.5px). `strokeCircle` rides along for free -- rx==ry is the EXACT-annulus case. Retired the `circleSegments` tessellation helper (now unused).
- **`strokeCircle` is now a true annulus** (exact SDF), not a clamped-inner-rim tessellation -- the inner-rim clamp is gone; once the half-thickness reaches the radius the annulus naturally fills to the centre.
- **Tests** rewritten to the SDF contract (6-vert quads, modes 4/5, half/cornerRadius fields, pad extents). **Verified on the 4090**: the yellow ellipse, the circle ring, and a new demo ellipse ring all smooth-edged at MSAA 1.

## 2026-06-14 тАФ L2: the transform stack -- push/translate/rotate/scale/pop ЁЯФДтЬЕ
- **`push()/pop()/translate(dx,dy)/rotate(rad)/scale(s|sx,sy)`** -- the ONE sanctioned piece of L2 drawing state, legal because it is explicitly scoped (push/pop pairs; `end()` asserts balance, `pop()`-without-`push()` throws). Completes the L2 spec's principle 3 (the principled answer to Processing's `rectMode(CENTER)`: to rotate a rect about its centre you `translate` to it, `rotate`, draw centred -- no hidden coordinate mode). **Verified on the 4090**: a rotated+scaled rounded-rect with a text label in the same nested transform; the rest of the frame unmoved (push/pop isolation).
- **Applied CPU-side at append time, to the POSITION only** (JOML `Matrix3x2f` current transform + an `ArrayDeque` stack). The SDF `local` and the `uv` stay in the shape's own frame -- they interpolate relative to the transformed quad like a UV, so the fragment computes its distance / samples its texel in that frame regardless of the transform. So the shader stays trivial and EVERY primitive (fills, strokes, SDF, image, text) transforms uniformly with zero per-primitive code. Identity transform = exact passthrough.
- **The one shader consequence:** the SDF (mode 1) edge AA switched from a fixed 1px `clamp(0.5 - d)` ramp to a **screen-space-gradient** ramp `clamp(0.5 - d/|grad d|)` (`length(dFdx(d), dFdy(d))`). At identity `|grad d| ~= 1` so it's the same as before (regression-checked on the 4090: un-transformed circles/rounded-rect look identical); under rotation/scale it keeps a clean ~1px edge. A nice side effect: resolution-independent SDF AA.
- **CPU-unit-tested** (unlike image/text, the matrix math needs no GPU): translate/scale/rotate transform points correctly, push/pop restores, unbalanced push caught at `end()`, pop-without-push throws. 36 tests green.
- **Known v1 limit**: tessellated shapes (ellipse, stroked rings) pick their segment count from the PRE-transform radius, so a heavily scaled-up ellipse can facet (the analytic SDF circle/rounded-rect don't). Catalogued.
- **Next**: the remaining post-v1 refinements (kerning, MSDF, combined `Style` fill+stroke, custom-`Font` loading), or a new altitude (a self-built GUI -- the natural first consumer of L2 + transforms).

## 2026-06-14 тАФ L2: multi-texture batching -- flush-on-texture-switch тЬВя╕ПЁЯЦ╝я╕ПтЬЕ
- **Lifted the one-image-texture-per-frame limit.** The vertex arena stays ONE buffer in paint order; it is now sliced into **draw runs** -- contiguous vertex ranges that each need a different texture bound. Flat/SDF shapes join the current run (they don't sample); an `image()` whose texture differs from the current run **opens a new run**. The Renderer issues one `vkCmdDraw` per run (with `firstVertex` offsetting into the shared arena), binding that run's texture -- **paint order preserved across any mix of shapes and images**. (raylib `rlgl`'s default-batch model, the spec's study reference.)
- **The Vulkan lesson that forced the design:** a single descriptor set **can't be rebound between two draws in one command buffer** -- the recorded draws would all observe the last write. The fix is one set PER RUN, from a **transient per-frame descriptor pool that is reset + re-allocated each frame** (the textbook "varying bindings per frame" pattern). One pool per frame in flight (resetting one never touches another in-flight frame's), fence-guarded like every per-frame resource; it grows if a frame ever needs more runs than it holds.
- **Run bookkeeping is CPU/unit-tested** (`runCount`/`runFirstVertex`/`runTexture`; a flat-only frame = one null-texture run); the run-SPLITTING is GPU-verified (needs real `Texture`s). The demo now draws **two distinct textures** -> two runs -> two draws, proving the switch.
- **Same-texture draws still merge** into one run (an atlas / sprite sheet is one draw); alternating textures costs a draw per switch -- the documented "group your draws" guidance, and the reason **text will use a single glyph atlas**.
- **Next: text** -- SDF glyphs (single-channel v1 / MSDF later) ride this exact textured path as **mode 3** (sample the atlas distance, smoothstep coverage), one atlas = one run. See [[L2 Feature Set - Renderer2D]].

## 2026-06-14 тАФ L2 strokes: the polygon outlines + MITER joins -- stroke set COMPLETE ЁЯУРтЬЕ
- **`strokeTriangle` + `strokeQuad`** via a shared **`strokePolygon`** -- the first place the **JOIN** question bites. Butt-capped edges would gap or overlap at corners; the fix is the **miter join**: offset each vertex along its angle BISECTOR (= normalized sum of the two adjacent edge normals) by `halfThickness / cos(phi)`, where `phi` is the angle between bisector and edge normal. The `1/cos` is the whole trick -- it makes the offset EDGES land exactly `halfThickness` from the originals (a plain offset along the bisector would pull corners in). Done for +/-bisector -> outer/inner corner points; the band between consecutive corners is two triangles. **Winding-agnostic** (symmetric +/-), so vertex order doesn't matter.
- **Miter limit**: a true miter spikes to infinity as a corner gets sharp (`cos -> 0`); `cos` is clamped to 0.1 -- a cheap cap. Real engines (SVG/Skia) fall back to a bevel past a limit; catalogued as a later refinement (SDF strokes reopen it).
- **Verified on the 4090**: thick triangle + quad outlines, corners **sharp, gap-free, uniform width through the join** (the user noticed -- the abstraction is one local per-vertex computation that serves any polygon). The square-`strokeQuad` test pins the math: a 100x100 square outline miters to outer bounds exactly `[-5, 105]`.
- **The v1 STROKE SET is complete**: `line`, `strokeRect`, `strokeCircle`/`strokeEllipse`, `strokeTriangle`/`strokeQuad` -- all CPU-triangulated, no `wideLines`, all feeding the one shape batch.
- **Next**: **SDF edge-AA** (the catalogued AA track for L2 curves/strokes -- smoothstep alpha over ~1px of signed distance), then the gated `image`/`text`.
