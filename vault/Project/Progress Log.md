## 2026-06-13 — Ring 3: the effect-contract guard (SPIRV-Cross reflection) 🛡️✅
- **Guard (c), the last of the three** ([[ShaderEffect - The Shadertoy Altitude#The contract, now ENFORCED (Ring 3 guard) ✅|the gap is closed]]): `ShaderReflection` enforces the v1 effect contract at creation via **SPIRV-Cross** (`lwjgl-spvc`). ShaderEffect was jvre's first "outside input" surface, and the contract had been only *documented* — a shader that COMPILES but breaks it would detonate deep in Vulkan, or silently with validation off. Now it fails fast, in the user's terms.
- **Reflects the SAME optimized SPIR-V jvre runs** (not a re-compile) — so the check matches exactly what the pipeline executes; if the optimizer drops an unused resource, the check drops it too, consistently.
- **Two rules, chosen to be optimizer-robust AND to catch the real crash cases:** (1) **no descriptor-bound resources** (the effect pipeline has no set layout → any UBO/sampler/image is a guaranteed mismatch; binding decorations are interface, survive optimization); (2) **push block ≤ 20 bytes** (jvre fills exactly its 20-byte range; bigger → reads past it). No block at all is allowed. Member-name/offset checks and "never writes outColor" deliberately deferred (the size check covers the crash; a blank screen isn't one).
- **SPIRV-Cross over hand-parsing SPIR-V** (the decision): the guard's job is to be *trustworthy*, and binary reflection isn't the mechanism worth hand-rolling here — safety wins. The same reflection is the **foundation for the post-v1 `set(name, value)`** uniforms (the twofer that justified building it now).
- **Tests** ([[Testing and CI-CD]]): two more GPU-free tests (clean effect + no-push effect pass; sampler-binding + oversized-push are rejected with named errors), doubling as proof the spvc natives load. Full suite 7 green. Verified the real `ripple.frag` still passes the guard and the app reaches the render loop.
- **The "what breaks jvre" plan is now fully executed** — all three rings have their guard (Ring 1 was already loud-and-early; Ring 2 = [[Diagnostics and the Crash Log]]; Ring 3 = this). Next: the bigger fork — **L2 Renderer2D batching**.

## 2026-06-13 — Diagnostics: the crash log (Ring 2 guard) 🩺✅
- **Guard (b) from the "what breaks jvre" discussion** is built ([[Diagnostics and the Crash Log]]): a `Diagnostics` log that captures an **environment fingerprint** to a per-OS app-data file the user can attach to a bug report. The reframe that shaped it: a Ring 2 fault (a hand-rolled-Vulkan bug) is *environment-dependent by definition*, so the payload is "what machine, configured how?", **not** a per-frame trace.
- **The fingerprint makes the never-run `CONCURRENT` branch self-reporting** — exactly the latent risk we named. The 4090 shares families (→ EXCLUSIVE), so the log now prints `sharing EXCLUSIVE (graphics=present=0)`; a split-family GPU would log `CONCURRENT (...)` and announce it's in the untested branch. Verified on the 4090: loader `1.4.350` + `validation: ON`, `NVIDIA GeForce RTX 4090 [discrete]`, `vendor=0x10DE device=0x2684 apiVersion=1.4.329 driverVersion=0x94D84000` (vendor-encoded → logged raw in hex, no wrong decode), format/present/sharing all captured.
- **The tee, not a rewrite**: `init` installs a two-destination `OutputStream` (decorator over `System.out`/`System.err`) writing to console AND one shared file — so every *existing* `System.out.println` (GPU line, swapchain line, validation messages) is captured for free, zero call sites rerouted. Enrichment was then just a few *richer* prints (loader+validation in Instance, type/vendor/api/driver in Device, the explicit EXCLUSIVE/CONCURRENT line in Swapchain).
- **A deliberate static singleton** — stated outright as a choice (against jvre's strict-ownership grain): `Vk.check` and the validation callback are both static, so a global sink is the standard for this cross-cutting concern.
- **Eager + flushed** is the cardinal rule: the fingerprint is written + flushed at startup because it's most valuable when the process dies hardest; `autoflush=true` flushes after every println so the last line survives a hard crash. Hot loop stays silent (events, not frames).
- **Frictionless MANUAL send** (decided over auto-upload): on a fault, `reportFault` prints the log path + a GitHub issue link — one click, consent preserved, no server to run. Auto-upload deferred *on purpose*: a consent/infra problem (the log path contains the username), not a difficulty one. **Verified end-to-end** — a deliberately bad shader path threw, and the log captured the full startup fingerprint + the FATAL stack trace + the "attach this file" footer. Per-OS dir (`%LOCALAPPDATA%` / `~/Library/Logs` / `~/.local/state`), previous run kept as `jvre.log.1`. Failure-tolerant: a missing/unwritable file degrades to "no log," never an exception.
- **Next: guard (c)** — the [[ShaderEffect - The Shadertoy Altitude#The known gap|Ring 3 SPIR-V reflection]] (enforce the effect contract at creation; doubles as the `set()` foundation). Its errors will flow through this sink for free.

## 2026-06-12 — Design: game-engine capability boundary + 3 Renderer constraints 🧭
- New note: [[Game-Engine Capabilities (planned)]] — sparked by "could a future game engine on jvre do animated pixel art?" Generalized to: *what must L1 expose so any game engine can be built on it?*
- **The boundary rule (settled):** **mechanism vs policy** — jvre owns any capability a higher layer physically can't synthesize from outside Vulkan/the window; the game engine owns anything composable from jvre's public surface. The teeth: keeping a feature *out* obligates jvre to expose the **seams** it's built from — e.g. `image()` MUST ship the sub-rectangle form, or sprite-sheet animation is impossible from above no matter how clever. Pixel-art mechanisms (Sprite/Animation/tilemap/timing) = game-layer; jvre just exposes textures + nearest sampling + alpha blend + UV sub-rect + (later) offscreen targets + batching.
- **Non-goals named** (so "anything a game engine wants" doesn't balloon jvre): audio, physics, networking, scripting, ECS/scene, asset-format parsing. jvre = render + present + input-surface only.
- **Tier-2 target catalogued** (implied by "any game engine," not yet roadmapped): present-mode/vsync, fullscreen, input, instancing, streaming/storage buffers, **offscreen render targets** (the gateway: pixel-perfect upscale, post FX, shadow maps), MRT, full pipeline state (depth/stencil/cull/topology/wireframe/blend), texture completeness (arrays/mips/cubemaps/compressed/sRGB), compute, multithreaded recording, HDR/color. All **L1 capability**; the game layer composes it.
- **Three boundary-case decisions:** **text IN** (policy, but absorbed — universal; already L2's `text()`); **camera OUT, primitives IN** (a 2D camera is pure policy = transform stack; jvre owns the *projection*, the game owns the *view* — `Camera2D` lives above jvre); **3D split** — L1 substrate stays **3D-capable** (nearly free; `mat4` + depth already there; you'd have to work to *prevent* it), L2 stays **flat by design** (Processing's 3D bolt-on is the cautionary tale), a friendly 3D API is a **deferred third altitude**.
- **Three constraints with deadlines** (shape code being written now — keep expressible, don't build): (1) **multithreaded command recording** — don't bake single-threaded pool ownership; (2) **offscreen render targets** — structure `drawFrame` as "render to a target (today = swapchain)," not one hardcoded swapchain pass; (3) **queue-set abstraction** — "queues by capability," not "the graphics queue" (cross-linked from [[Device Selection and Cross-Platform (planned)]]).
- Pure design session — no code. Reinforces [[API Vision - Layered Altitudes]] / [[Design North Star]]: all Tier-2 is the public-L1 half of the thesis.

## 2026-06-12 — VMA: the standing ticket closes, 6 -> 0 🧹 ✅
- **The memory milestone** ([[VMA - Vulkan Memory Allocator]]): every best-practices allocation advisory -- accruing since the first vertex buffer -- is gone. stderr **fully silent** on the 4090; clean shutdown (vmaDestroyAllocator's leak assert = a free leak detector, passing).
- **Why**: drivers cap allocations (`maxMemoryAllocationCount` can be 4096), `vkAllocateMemory` is slow, and real engines SUB-allocate from few big blocks -- a library-sized problem; VMA (AMD) is the industry standard. LWJGL's `lwjgl-vma` (real natives, unlike lwjgl-vulkan).
- **Integration**: `Device` owns the `VmaAllocator` (device-scoped; created after queues, destroyed before the device -- the recreatable-context seam holds); `VmaVulkanFunctions` hands VMA the dynamically-loaded function pointers.
- **The conceptual shift -- intent over flags**: `Buffer(.., boolean hostVisible)` replaces memory-property flags; `vmaCreateBuffer`/`vmaCreateImage` collapse create/hunt/allocate/bind into one call. `Device.findMemoryType` retired with zero callers (concept stays in [[Vertex Buffers and GPU Memory]] + git history). Uploads now `vmaMap/vmaUnmap/vmaFlush` (flush no-ops on coherent -- correct on ANY memory VMA picks).
- **Attachment exception**: the depth image gets `DEDICATED_MEMORY_BIT` (large, long-lived, resize-recreated, driver-optimizable) -- pooling is for the many-small case; MSAA's color target will be the same.
- **Migration invisible by design**: only the memory half changed; cube renders pixel-identically. Empirical arc: 6 warns -> 1 (Buffer done) -> 0 (images done).
- **Next: MSAA** -- all prerequisites (own-VkImage, depth, dedicated-attachment idiom) now in hand.

## 2026-06-12 — Design: L2 feature set formalized (draft) 📐
- New note: [[L2 Feature Set - Renderer2D]] — principles + v1 primitive set for the "just draw" altitude, coined on Processing but deliberately smaller.
- **Headline decision (owner's call, confirmed): NO declaration modes.** Processing's `rectMode(CENTER)`-style switches are rejected -- global mutable state, action at a distance, composes badly; the user-side translation is one self-documenting subtraction. Refinements: one *natural* convention per shape (rect = corner+size, circle = centre+radius -- no single global rule), and the real demand behind CENTER mode (rotation about the centre) is answered by the **transform stack** (`push/translate/rotate/pop`, the one sanctioned state because it's scoped), not by declaration modes. Stateless escape if ever needed: a distinct name (`rectCentered`), never a mode.
- Also pinned: pixels/top-left/y-down (no NDC at L2), immediate mode (fits per-frame recording exactly), L2-speaking errors, per-call color as default position. Open questions logged in the note: per-call color vs Style object, stroke variants in v1, `rect` vs `fillRect` naming.
- **All three settled same day**: per-call color confirmed; **outlines are v1 citizens** (triangulated CPU-side -- `wideLines` is non-portable); naming = **symmetric `fillRect`/`strokeRect` pairs** for closed shapes, bare `line`/`text`/`image` for one-form primitives ("stroke" is the domain's word; awt's `drawRect` ambiguity rejected). Fill+stroke are **orthogonal** (bordered = two calls in v1); the combined form is reserved for the `Style` general form -- motivated by a *correctness* need, not sugar: centered strokes overlap the fill, and with alpha, draw-twice produces two different blends within one stroke; only disjoint-geometry-in-one-op renders it right. **The v1 surface is settled.**
- Also assessed (2026-06-12, conversation): **Vulkan 1.4** -- no refactor warranted. It's a consolidation release (push descriptors, dynamic-rendering local read, optional host image copy go core); our 1.3 path (dynamic rendering + sync2) remains the modern default, and the UHD 620's driver tops out at 1.3, so raising the floor would break the craptop. Revisit per-feature via extensions when needed (`VK_KHR_push_descriptor` at the descriptor milestone, host image copy at textures).
