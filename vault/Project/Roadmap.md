# Roadmap (forward-looking)

**As of 2026-06-16 -- jvre 1.0.0 is RELEASED on Maven Central** (`io.github.milquetoad:jvre:1.0.0`). The backward diary is [[Progress Log]]; the original bootstrap checklist is [[Roadmap - Clear to Color]]. This note is now the *post-1.0 forward plan* (Phase 3 remaining + Phase 4) and the *why* behind its ordering.

## Where we are -- 1.0 shipped 🏁
The full Vulkan substrate (L0/L1 plumbing), **both API altitudes, AND the L1 escape hatch** are delivered, documented, and released:
- **L2 `Renderer2D`** -- fills, strokes, the SDF render path (incl. crisp SDF curves), image + multi-texture batching + **file loading** (`loadImage`) + per-texture **`Filter`**, text via SDF glyphs + measurement, the transform stack. See [[L2 Feature Set - Renderer2D]].
- **`ShaderEffect`** -- the fullscreen Shadertoy altitude, contract-guarded.
- **L1 escape hatch** -- a sanctioned "bring your own geometry + shaders + uniforms + textures" surface (`PipelineSpec`/`VertexLayout`/`FrameRenderer`/`SceneRenderer` + `Camera`), mixable with L2; `Device` exposes raw Vulkan. The dogfood (the cube via the public API) retired the hardcoded SCENE.
- **Interactivity + capability knobs** -- per-frame `Input` snapshot, `time()`/`dt()`, creation-time `RendererOptions` (vsync/MSAA/GPU).
- **Delivery** -- cross-platform CI (Win+Linux), de-pinned natives, GPG-signed Maven Central release, a documented + **audited** public surface ([[Public API surface|docs/api-surface.md]]), user guides in `docs/`.

Measured against the [[Design North Star]]: **both halves delivered** -- approachability (L2 + defaults + the "hello rectangle" docs) AND flexibility (the L1 geometry escape hatch). The standing North Star debts from the 0.x era are paid. The forward work is now **building jvre out to FULLY-FLEDGED** (see [[jvre-fully-fledged-not-pull-based]]).

## Prioritization lens (why this order)
1. **On-philosophy first.** The North Star's two explicit debts -- *prove L2 approachability* and *deliver L1 flexibility* -- outrank net-new capability.
2. **Cheap + high-leverage before big + speculative.** Small seams that unlock whole categories (input, time) come first.
3. **Honor forcing functions.** The owner is migrating to Linux; cross-platform is a first-class constraint that has *never been verified* -- validate it before more code piles on top.
4. **~~Refinements are pull-based.~~ SUPERSEDED 2026-06-16 -- jvre will be FULLY-FLEDGED.** The owner has committed to building out the *complete* expected feature set of a serious rendering framework (the bigger of "rendering layer of a game engine" or "rendering software"), as **planned, committed work** -- NOT "ships when a consumer needs it." This lands **post-1.0** (1.0 is still phases 1-2 + the release track), but the catalogued refinements (mipmaps/anisotropy, kerning, MSDF, `Style`, custom fonts, blend modes, ...) and the power axis below are now a committed backlog to schedule, not opportunistic polish. A dedicated post-1.0 feature-plan pass is owed. See [[jvre-fully-fledged-not-pull-based]] (memory).
5. **The power axis is committed, not "if need arises."** Compute / ray tracing / instancing / render-to-texture are genuine planned milestones (post-1.0), part of being fully-fledged -- still sequenced after the standing debts + 1.0, but no longer framed as merely "reachable, not promised."

## Phase 1 -- Make L2 interactive, and prove it (near-term)
The cheapest, highest-leverage work, and it pays both North Star debts on the L2 side.

- **1a. Input seam.** An L2-friendly per-frame input snapshot: mouse position + buttons, scroll, keyboard, typed chars. Today only `Window.cursorPos` is exposed, and in raw LWJGL form. *Why first:* L2 can draw but not react -- this is the single biggest functional gap, it unlocks every interactive sketch, and it's the prerequisite for the GUI demo. *Size:* small-medium. *Design:* snapshot polled in the frame loop, surfaced through the Renderer/L2 (not raw GLFW callbacks leaking up).
- **1b. Time / delta source.** The Renderer already tracks frame time internally; expose `time()` / `dt()` so animation is first-class at L2 (today a user must reach for `System.nanoTime()`). *Size:* trivial. *Depends:* none.
- **1c. Cross-platform validation + minimal CI.** Flip `lwjglNatives` and actually run on Linux; stand up GitHub Actions to build + run the GPU-free unit tests on Windows **and** Linux. *Why now:* the migration is the forcing function; a broken Linux build is cheapest to catch early, and CI stops it silently rotting. *Size:* small-medium. *Depends:* none (do in parallel).
- **1d. Approachability proof + first user docs. ✅ MOSTLY DONE (2026-06-16).** Wrote the user-facing doc set in `docs/` (getting-started with the literal "hello rectangle", a full 2D-graphics guide, shader-effects + custom-pipelines guides, an overview/map) and refreshed the README (killed "early days", added an honest **scope statement** -- not a game engine / not a GUI toolkit -- a status+capabilities section, and a docs index). Writing the docs *was* the first-user pass: it surfaced the missing **image loading** (queued next) and confirmed the zero-config path (`RendererOptions.defaults()`). **Still open (folded into the release-track readiness):** a dedicated L2-language error-message audit and 1-2 standalone example sketches beyond `Main`/`GuiDemo`.

## Phase 2 -- Deliver L1 flexibility (mid-term)
The most *vision-significant* gap: the North Star promises dropping to custom shaders/geometry "without leaving the engine," but only fullscreen effects honor it today.

- **2a. The geometry escape hatch. ✅ COMPLETE (2026-06-15).** A sanctioned path to render user geometry + shaders + uniforms + textures, mixed with L2 -- the keystone of L1, the North Star's flexibility half delivered.
  - Beat 1: user-defined pipelines via the thin guarded seam (`PipelineSpec`/`createPipeline` + `SceneRenderer`/`FrameRenderer`), additive through `Pipeline.Kind.CUSTOM`.
  - Beat 2: index buffers; bound resources (UBO/texture/push, jvre-owned descriptors); the **Camera (2c done)**; and the **dogfood** -- the engine's textured cube reproduced via the public API, then the hardcoded `SCENE` retired (no built-in geometry left). See [[L1 Escape Hatch]].
- **2b. Capability knobs. ✅ COMPLETE (2026-06-15).** A creation-time `RendererOptions` config carrying: **vsync** (FIFO vs MAILBOX), **MSAA** sample count (1/2/4/8, incl. the no-resolve off path), and **GPU-selection override** (`preferGpu` name substring over the default scoring; runtime switching still future). Clears the hardcoded MAILBOX/MSAA/GPU TODOs. See [[Device Selection and Cross-Platform (planned)]], [[Game-Engine Capabilities (planned)]].
- **2c. A camera helper. ✅ DONE (2026-06-15, with 2a).** An *on-demand* view + projection helper that centralizes the Vulkan-correct projection -- the JOML Y-flip + `zZeroToOne` clip-space lesson already solved once for the cube -- so 3D users don't re-derive it (hide the Vulkan gotcha, not the concept). **Matrix math only in v1** (controllers like orbit/FPS are input-bound -> examples, not the helper). *Open decision:* expose JOML `Matrix4f`, a neutral `float[16]`, or both (JOML is currently implementation-only; a public Camera would surface it). *Why here:* turns "render a mesh" (2a) into "render a scene" -- the API vision lists Mesh/**Camera** together.

## Phase 3 -- Pull-based polish & the first real consumer
Slot opportunistically; let need drive it.

- **3a. L2 refinements.** **SDF curves ✅ DONE (2026-06-15).** `fillEllipse`/`strokeCircle`/`strokeEllipse` moved from tessellation to the SDF path (crisp at any size, MSAA-independent): `strokeCircle` is an exact annulus SDF, the ellipses use iq's *approximate* ellipse distance (good enough for AA). One 6-vertex quad each (modes 4/5), `sdfBox` generalized to `sdfQuad`, the tessellation helper (`circleSegments`) retired. **Image loading ✅ DONE (2026-06-16)** -- `renderer.loadImage(path)` decodes PNG/JPEG/... via stb_image (the gap the docs surfaced). **Sampler filter ✅ DONE (2026-06-16)** -- a `Filter` enum (NEAREST/LINEAR) per texture; defaults by intent (createImage->NEAREST, loadImage->LINEAR). (Filtering changes artifact *character*, not detail -- no super-resolution.) Remaining refinements (now **committed**, not pull-based -- see the prioritization lens): the rest of sampler config (**mipmaps + anisotropy**, the minification case), kerning + MSDF (text), the combined `Style` fill+stroke (the translucent-overlap *correctness* case), blend modes. **Custom-font (TTF) loading is scheduled at [4b](#phase-4----the-power-axis-longer-term)** (right after render-to-texture, per owner request).
- **3b. A tiny immediate-mode GUI demo. ✅ DONE (2026-06-15).** A worked example built ON L2, **not** a jvre feature -- it lives in `jvre.demo` (excluded from the published jar), like `jvre.tools`. `Gui` (label + button + slider) over the **hot/active-ID** core; `GuiDemo` drives a bouncing ball (`gradlew runGuiDemo`). As a real API stress test it surfaced two missing L2 methods -- **`textWidth` + `lineHeight`** (text measurement for layout) -- now added to `Renderer2D` (the pull-based-polish loop working as designed). See [[Self-Built GUI (planned)]].

## Release track -- toward v1.0 (runs alongside Phases 1-2)
The project-management / delivery half of the learning goal. Climbs a ladder rather than jumping straight to Central.

- **R1. De-pin the natives (blocking prerequisite).** `build.gradle` currently pins `lwjglNatives = 'natives-windows'`; a consumable library must let the *consumer* pick their platform's natives (LWJGL's standard `api`-deps + per-OS classifier pattern). Nothing ships until this is fixed. Pairs naturally with cross-platform (1c).
- **R2. Publishing plumbing.** Gradle `maven-publish`: the main JAR + **sources** + **javadoc** jars, a complete POM (name/description/license/SCM/developer), semver `0.x` while the API churns.
- **R3. Pre-1.0 distribution.** Tagged **GitHub Releases** + **JitPack** (zero-setup consumption from a git tag) so others can try it without Central's bureaucracy. Wire release builds into CI (extends 1c).
- **R4. Maven Central at 1.0.** Verified namespace **`io.github.milquetoad`** (GitHub-verified, free), **GPG-signed** artifacts, the sources/javadoc jars, via the Central Portal. Done once the API is stable -- re-publishing breaking `0.x` to Central is painful and discouraged.
  - **API surface audit ✅ DONE (2026-06-16)** -- the "API declared stable" precondition: pruned `jvre.core` 29 -> 26 public types (hid `Vk`/`Commands`/`Swapchain`, trimmed `Buffer`/`Texture`/`Pipeline`/`Font` to opaque handles, kept `Device` as the raw escape). Principle: tighten now (reversible), widen later (additive). Contract written down in `docs/api-surface.md`.
  - **Namespace verified + GPG key created ✅ (owner, 2026-06-16).**
  - **Build plumbing ✅ DONE (2026-06-16):** `signing` plugin (opt-in `-PsignArtifacts`, `useGpgCmd`), a local staging repo + `centralBundle` Zip task producing `build/central-bundle-<version>.zip` (the manual-bundle path -- transparent, no 3rd-party publish plugin). Unsigned dry-run verified the layout + checksums.
  - **Remaining R4 (owner):** add `signing.gnupg.keyName` + the Central Portal token to `~/.gradle/gradle.properties`, run `gradlew centralBundle -PsignArtifacts`, upload the bundle (Portal UI / Publisher API), validate. **Then the version call: 1.0.0 now vs a 0.2.0 Central shakedown first.**
- Cross-cutting: a **getting-started + API overview + examples** doc set (shares work with 1d), and an **honest scope statement** in the README.

## Phase 4 -- The power axis (longer-term)
Net-new capability, post-1.0. Per the fully-fledged direction these are **committed**, not "reachable, not promised." (Several spitballed 2026-06-15; sequenced here.)

- **4a. Render-to-texture / offscreen targets. ✅ DONE (2026-06-16).** Render into an image instead of the swapchain -- the FOUNDATION for readback and post-processing / effect chains. `RenderTarget` (a sampleable color image + optional MSAA + depth) that *is-a* `Texture`; the render loop is now target-agnostic (queued target passes before the swapchain pass, the swapchain just the last target). Both altitudes: **L1** `drawToTarget(target, frame -> ...)` + **L2** `createCanvas(target)` (a `Renderer2D` into the target -- the Processing `createGraphics` analog, which drove the per-surface `ShapeBatch` refactor). A target matches the swapchain's format/depth/samples so every baked pipeline accepts it, and inherits the renderer's MSAA (render -> resolve into the sampleable image). Full write-up + the deferred refinements (1x-target-on-MSAA-screen, `target.resize`, format-change rebuild): [[Render to Texture and Offscreen Targets]]. **4d (readback) + 4e (shape mask) now have their foundation.**
- **4b. Public font loading (TTF). ✅ DONE (2026-06-17).** A sanctioned custom-font API -- `renderer.loadFont(resourcePath)` / `loadFont(resourcePath, pixelHeight)` returning a `Font` -- so users aren't limited to the built-in DejaVu. The mirror of `loadImage`: `Font.load` (stb_truetype SDF bake) + `text(Font, ...)` already existed, so this was purely additive -- a public creation path + a public `Font.close()` (a loaded font is caller-owned; the built-in `font()` stays renderer-owned) + a `lineHeight(Font, float)` parity overload. The 2026-06-16 audit's "tighten now, widen later" paid off exactly. Verified on the 4090 (2nd bake logged, renders, validation silent, clean teardown).
- **4c. Rect / scissor clip (L2). ✅ DONE (2026-06-17).** `g.pushClip(x,y,w,h)` / `g.popClip()` -- restrict drawing to a rectangle (scroll views, UI panels, masked regions). A clip stack alongside the transform stack, driven by the **dynamic scissor** (`vkCmdSetScissor`). As predicted, a clip change flushes the batch at that boundary -- generalized the draw RUN to carry `(firstVertex, texture, clip)`, with a shared `ensureRun()` splitting a run on either a texture switch OR a clip change. The clip rect is the given rect's transformed axis-aligned bounding box, intersected with parent + framebuffer (translate/scale exact; rotation -> bounding box, since a scissor can't rotate -- tight rotated/non-rect masking is **4e**). Verified on the 4090 (validation silent, clean teardown). Lesson: `ArrayDeque` forbids null, so the null parent-clip needed a `LinkedList`.
- **4d. Headless + frame readback. ✅ DONE (2026-06-17).** Render with no window/swapchain/display -- offscreen into RenderTargets + read the pixels back. The payoff: automated visual-regression testing + offscreen image generation.
  - **Readback ✅** -- `renderer.readPixels(target)` -> RGBA8 `byte[]` (image -> host-readable staging via `Commands.oneShot`; `Buffer.HostAccess.READ`/`readback`/`downloadBytes`; `Texture.renderTarget` gained `TRANSFER_SRC`; BGRA->RGBA swizzle). Fixed a latent validation WARN (READ src access mask on the SHADER_READ_ONLY->TRANSFER_SRC transition).
  - **Headless ✅** -- surface-optional `Device` (no present/swapchain when `surface == null`), a headless `Instance` (no GLFW surface exts / no `glfwInit` -- LWJGL loads Vulkan independently), and a swapchain-optional `Renderer` (headless ctor + `render()`, a synchronous one-shot reusing the WHOLE target-pass machinery; only the present loop absent). Formats from `RenderFormats.headless` (the keystone seam paid off). `HeadlessDemo` / `gradlew runHeadless` writes a correct PNG with no window. Verified on the 4090, validation silent.
  - **Remaining (separately catalogued):** wiring the golden-PNG regression test into CI needs a GPU runner or a SOFTWARE Vulkan impl (lavapipe/SwiftShader) -- the current CI is GPU-free. The capability is done; the CI plumbing is its own task.
- **4e. Shape / mask clip (L2).** Clip to a non-rectangular region (circle, polygon, another shape) -- Canvas/Skia `clip(path)`. Via a **stencil mask** (draw the mask to the stencil buffer, then stencil-test the content) or render-to-texture + alpha mask. *Builds on 4a + stencil support.* The richer 2D follow-on to the rect clip (4c).
- **4f. Instanced rendering.** One draw call -> N copies with per-instance data (particles, tiles, many meshes). A real Vulkan lesson (per-instance input rate, instance buffers, `vkCmdDraw` instanceCount). Presupposes the 2a geometry path; jvre's L2 already covers "many shapes" via the vertex arena, so this is the 3D/scene side.
- **4g. GPU timestamp queries.** `vkCmdWriteTimestamp` + query pools to measure GPU-side pass timing -- profiling + a perf HUD. Extends the [[Diagnostics and the Crash Log|Diagnostics]] story from "what crashed" to "what's slow."
- **4h. 3D user clip planes.** Arbitrary clip planes in 3D (`shaderClipDistance` / `gl_ClipDistance`) -- cross-sections, "clip below the water plane" reflections, CAD cutaways. Specialized 3D/L1 item, gated behind enabling the device feature. (Frustum near/far clipping is already inherent in the projection -- not a feature.)
- **4i. Compute shaders** -- compute pipeline, dispatch, storage buffers/images. The big "powerful" milestone the discrete-GPU substrate makes reachable.
- **4j. Ray / path tracing** -- the long-horizon aspiration; reachable on the hardware.

## Phase 4 -- Feature batches (owner triage, 2026-06-17)
A batch of fully-fledged features requested 2026-06-17, parsed + grouped by subsystem, size (S/M/L), and -- the load-bearing part -- **shared prerequisites**, so a refactor isn't paid twice. Per [[jvre-fully-fledged-not-pull-based]] these are committed work. Sequence: **Batch 1 -> Batch 2 + 3-kerning -> the Keystone + headless + Batch 5 -> Batch 4 + the MSDF spike -> Batch 6 (channel & format expansion).** Batches K, 1-5 are DONE; **Batch 6 is the next committed work** (catalogued 2026-06-18).

- **Batch K -- the format keystone. ✅ DONE (2026-06-17).** Decoupled color/depth/sample formats from the swapchain into `RenderFormats` (a small internal record). The Renderer holds one (sourced from the swapchain via `fromSwapchain`, refreshed on recreation), and every bake site (`buildShapePipeline`/`buildEffectPipeline`/`createPipeline`/`createRenderTarget`) reads from it, not the swapchain. A behavior-preserving refactor (verified: demo identical, validation silent) whose value is the SEAM: **headless** (supply default formats, no swapchain), **float/HDR targets** (override the color format), and **per-target pipeline variants** now all plug in here without touching the bake sites again.

- **Batch 1 -- OS / window surface. ✅ DONE (2026-06-17).** Five small, independent, GLFW-backed `Window`/`Input` features: `setTitle` (runtime title); `setCursor(CursorShape)` (standard OS cursors, cached); `clipboard()`/`setClipboard()`; `Input.droppedFiles()` (a `GLFWDropCallback` -> per-frame dropped-paths list); HiDPI -- `contentScaleX/Y()` + `windowSize()` exposed, with the framebuffer-px-vs-window-coords story documented (the L2 pixel space already = framebuffer px, so drawing is DPI-correct without a multiply). New public `CursorShape` enum. `setTitle` auto-verified live; validation silent.

- **Batch 2 -- sampler / texture completeness. ✅ DONE (2026-06-17).** The catalogued "sampler config." A new `TextureOptions` builder bundles the four sampler knobs (one param, not an overload thicket). **Wrap** -- `WrapMode` enum (CLAMP/REPEAT/MIRROR/BORDER) -> the sampler addressMode. **Mipmaps** -- GPU-side `vkCmdBlitImage` pyramid generation (+ `TRANSFER_SRC` usage, whole-chain view, `MIPMAP_MODE_LINEAR` sampler, a `SAMPLED_IMAGE_FILTER_LINEAR` format guard). **Anisotropy** -- the predicted cross-layer coupling: `Device` now enables the optional-core `samplerAnisotropy` feature + remembers `maxSamplerAnisotropy`, and the sampler requests it (clamped). Verified on the 4090 (validation silent through the mip-gen blit chain + the feature path; minification payoff subtle on the small demo cube).

- **Batch 3 -- text refinements.** **Kerning ✅ DONE (2026-06-17).** Pair-kerning precomputed at bake (`stbtt_GetCodepointKernAdvance` -> a `float[]` table on `Font`), applied in `text()`/`textWidth()` (pen nudged per glyph pair). stb reads only the LEGACY `kern` table (not GPOS -- a shaper is out of scope); a bake-time diagnostic confirmed DejaVu carries 220 ASCII kern pairs, so it kerns. Verified on the 4090 (validation silent). **MSDF ✅ DONE (2026-06-18).** Spike + implementation in one. See [[MSDF Text - Spike Findings]]. The old premise ("no LWJGL binding -> bundle offline atlases") was **STALE**: `org.lwjgl:lwjgl-msdfgen` exists (a binding to Chlumsky's msdfgen), a standard LWJGL module with natives, published since **3.3.4** (jvre's exact version) -- so it was added with **no version bump**. Shipped as an **OPT-IN second glyph path parallel to the default single-channel SDF**: `Renderer.loadMsdfFont` bakes a `Font` whose RGBA8 UNORM atlas holds three distance fields generated at runtime (stb_truetype outline -> msdfgen `Shape` of linear/quadratic/cubic segments -> normalize/orient/edge-colour -> `generate_msdf` -> quantize). `Font.bake` now shares the metric/packer/UV/kern skeleton between SDF and MSDF, branching only on the per-glyph bitmap + atlas format; `shape2d.frag` gained mode 6 (median-of-3 + `screenPxRange`). SDF stays the default (lighter R8, fine for UI sizes); MSDF is the upgrade for large display text. **Verified headless** (`runMsdf`: same font baked both ways at 48px, rendered at 120px -- MSDF apexes crisp where SDF rounds; build green, validation silent). Two bugs found + fixed en route: a heap-corruption from a bulk `ByteBuffer.get` advancing the position so `stbtt_FreeSDF` freed the wrong pointer (the shared SDF path), and msdfgen's y-UP bitmap needing a vertical flip to match the y-DOWN atlas. See [[l2-text-sdf-glyphs]].

- **Batch 4 -- shader authoring workflow. ✅ DONE (2026-06-18).** **Structured compile diagnostics:** `ShaderCompiler` now parses shaderc's log into typed `ShaderDiagnostic` records `(name, line, severity, message, raw)` and throws `ShaderCompileException` (still a `RuntimeException`, so behaviour is preserved) carrying `diagnostics()`/`errors()`/`rawLog()`. Parser is best-effort + lossless (unrecognized log -> one UNKNOWN diagnostic, never an empty list); a unit test pins the real shaderc format on this toolchain. **Pipeline/shader hot-rebuild:** `Pipeline.reloadShaders(byte[]|String x2)` re-bakes the VkPipeline IN PLACE (same object, so a captured scene-renderer lambda keeps working) from new SPIR-V/GLSL -- same formats + fixed-function + resource interface; only the shader body changes. Drains the GPU first (not hot-path); a failed compile throws and leaves the running pipeline intact (the live-reload story). Implemented by baking a fresh sibling through the normal path and adopting its handles (no duplication of the 400-line bake). Verified headless (`runReload`: render red -> `reloadShaders` -> render blue, same pipeline; output changed, validation silent). **ShaderEffect live-reload ✅ DONE (2026-06-18):** `setEffect` is now safe to call again mid-loop -- when an effect is already installed it swaps the new fragment shader into the existing pipeline IN PLACE via the hot-rebuild hook (drains first), instead of the old close-then-rebuild that could free a pipeline still in flight. The new `ShaderEffect` re-compiles + re-runs the effect-contract guard at creation, so a broken edit throws there and the running effect keeps going. Demo wires it to F5 (edit `ripple.frag`, press F5). Dogfoods the Batch 4 reload hook; closes Batch 4 entirely. (The swapchain-format-change rebuild for engine-owned pipelines still recreates them from scratch in `recreateSwapchain`; the hook could unify that too, later.)

- **Batch 6 -- channel & format expansion (owner triage, 2026-06-18).** Four requests, parsed + scoped against the now-shipped channel + format machinery. Builds directly on Batch 5 (iChannels) and Batch K (the format keystone). Sequence within the batch: **6a (trivial, additive) -> 6b + 6c (share the channel-dimensionality refactor) -> 6d (smallest given 6b/6c land).**
  - **6a. 32-bit float render targets. ✅ DONE (2026-06-18).** Added `TargetFormat.HDR_FLOAT32` (`R32G32B32A32_SFLOAT`) alongside the existing 16-bit `TargetFormat.HDR`. Purely additive -- the Batch-K keystone already routes `format.vk` generically through `createRenderTarget` -> `Texture` -> per-target pipeline baking, and the `readPixels` guard already rejects any non-8-bit format -- so the enum constant + its doc was the whole change. **Decision (2026-06-18): keep BOTH, don't replace.** 16f is the display/post-processing format (half the bandwidth, matches HDR swapchains -- the right default for tone-map/bloom); 32f is the precision/data format (accumulation buffers for progressive path tracing, position/normal G-buffers, exact `readPixels`). They are not redundant. **Additive-only constraint:** `TargetFormat.HDR` is public API as of 1.1, so the 32-bit option is a NEW constant (e.g. `HDR_FLOAT32`/`FLOAT32`, name TBD at impl), not a rename. The Batch-K keystone already decoupled color format from the swapchain, so this plugs in at the bake sites with no new refactor. *Size: S.*
  - **6b. Cubemap channels (`samplerCube`).** Extend the channel system (iChannels + L1 texture channels) to cube maps -- environment/reflection probes, skyboxes, the Shadertoy cubemap-channel convention. Real Vulkan lesson: a cube-compatible image (`VK_IMAGE_CREATE_CUBE_COMPATIBLE_BIT`, 6 array layers) + a `VK_IMAGE_VIEW_TYPE_CUBE` view + a new `Texture` creation path (6 faces). **Shared prerequisite with 6c (the load-bearing part):** `ShaderReflection` today keys only off "sampled image at set 0, binding 0..3" -- it does NOT read the sampler's *dimensionality*, so a `samplerCube` would PASS the contract guard but get a 2D view bound to it -> validation error. So 6b+6c both need the reflection to read the SPIR-V `Dim` (2D/Cube/3D) and the renderer to bind a *matching* view (+ validate the bound texture's kind against the declared sampler). Pay that refactor once. *Size: M-L.*
  - **6c. 3D volume channels (`sampler3D`).** The volume analog of 6b -- `VK_IMAGE_TYPE_3D` + `VK_IMAGE_VIEW_TYPE_3D` + a depth extent. Volumetric data, 3D LUTs, raymarched fields. Shares 6b's reflection-dimensionality refactor; differs only in the image/view type and the upload (a 3D extent, not 6 faces). *Size: M (given 6b's refactor).*
  - **6d. Dynamic / CPU-updatable channel textures (the keyboard-channel decomposition, owner call 2026-06-18).** The original "keyboard channel" request, re-cleaved at the mechanism/policy seam. The engine TODAY uploads textures **once** (staging upload at creation -- no public per-frame CPU update path), so a keyboard texture (changes every frame) needs a NEW mechanism: a **dynamic texture** whose pixels can be rewritten each frame and bound as a channel. That primitive is the committed engine-side work here -- and it's reusable far beyond keyboard (audio FFT bins, procedural data, any per-frame CPU->GPU channel). *Size: S-M.*
    - **The Shadertoy keyboard layout is NOT core API -- it's a `jvre.demo` example.** Decided 2026-06-18 (owner): the 256x3 row convention (row 0 = down, row 1 = pressed-this-frame, row 2 = toggle) AND the GLFW-key -> JS/USB-keycode map are *policy imported from another platform*. Baking them into the core is the kind of policy-in-engine jvre avoids, and it's self-defeating -- the only reason to match Shadertoy's exact layout is verbatim shader portability, yet if the engine owns the keycode map a consumer can't correct it. Left app-side they just write the column they want. So the keyboard channel becomes a **worked example in `jvre.demo`** (excluded from the jar, like the GUI demo): it demonstrates 6d's dynamic-channel mechanism + `Input` + a keycode map, without committing either convention to the public surface. The [[Self-Built GUI (planned)|GUI-demo precedent]] is exactly this shape.
  - **NOT in this batch -- clipboard get/set is ALREADY DONE.** Requested 2026-06-18, but it shipped in **Batch 1 (2026-06-17)** as `Window.clipboard()` / `setClipboard()` (GLFW text clipboard), documented in `docs/2d-graphics.md`. Public already. (An image/arbitrary-data clipboard would be net-new + OS-specific -- not requested, not scoped here.)

- **Batch 5 -- RTT / effect power** (on Batch K). **Float/HDR targets ✅ DONE (2026-06-17):** `TargetFormat.HDR` (R16G16B16A16_SFLOAT) on `createRenderTarget` + `createPipeline(spec, target)` (per-target-format baking, the keystone's payoff); `readPixels` guarded to 8-bit. Demo'd as a tone-mapping chain (`runHdr`, clamp vs Reinhard split), owner-verified. **Bind N input textures as channels ✅ DONE (2026-06-18):** `PipelineSpec.texture(Stage)` is now ADDITIVE -- each call adds the next `sampler2D` channel at bindings 1, 2, 3, ... (was a single boolean `hasTexture`). `Pipeline` builds N descriptor bindings + `uploadTexture(frame, channel, tex)`; `FrameRenderer.texture(channel, tex)` writes a specific channel (`texture(tex)` = channel 0). One pass can now sample several targets at once -- the foundation of post-processing *chains*. Verified headless (`runChannels`: red x-gradient -> target A, green y-gradient -> target B, a 2-channel BLEND pass samples both -> C; centre pixel R188 G188 B0, blend PNG correct). **The "target samples target rendered black" lead is RESOLVED (2026-06-18):** it was a GENERAL `FrameRenderer` bug (descriptor set bound before `uniform()`/`texture()` wrote it -> black FIRST frame for every custom pipeline; effect chains, being always-first-use, broke every frame). Fixed by binding the descriptor at draw time. Relates to the [[Game-Engine Capabilities (planned)|MRT/post-processing + HDR rows]]. **ShaderEffect iChannels ✅ DONE (2026-06-18):** the multi-binding is now surfaced through the `ShaderEffect` facade as Shadertoy `iChannel0..3`. An effect declares `layout(set=0, binding=N) uniform sampler2D iChannelN;` (up to 4); the contract guard (`ShaderReflection`) was RELAXED to allow samplers at set 0 / binding 0..3 (rejecting UBOs, storage images, out-of-range samplers) and reflects the channel count via SPIRV-Cross binding decorations. The FULLSCREEN_EFFECT pipeline grew a per-frame sampler descriptor set (binding base 0, no UBO -- mirrors the custom path's machinery via a shared `textureBindingBase`); `Renderer.setEffectChannel(i, tex)` feeds each channel (unset = 1x1 default), written + bound per frame before the effect draw. Re-pointable any frame -> feed an effect a `RenderTarget.texture()` for a post-processing pass. Verified: reflection unit-tested (GPU-free -- samplers 0..3 accepted with right count, UBO + out-of-range rejected) + a windowed channel demo (`channel.frag` samples iChannel0 = the test image; validation silent through 14s of frames on the 4090).

## Deferred / non-goals (recorded so they don't creep back)
- **A GUI toolkit as a feature pillar** -- out of scope; jvre is a rendering framework (Processing, its upper reference, isn't a GUI toolkit either). A *bounded demo* (3b) is the only sanctioned form; if a real GUI is ever needed for tooling, drop in Dear ImGui / Nuklear ([[GUI Options]]).
- **Beginner/artist friendliness** (hidden `main`, no types) -- a deliberate non-goal per the North Star audience note; chasing it costs the flexibility that is jvre's edge.
- **Retained-mode anything** -- immediate mode is the paradigm throughout.

## The one-line ordering
**Interactivity + proof (1) -> L1 flexibility (2) -> [v1.0: ship to Maven Central] -> pull-based polish + a GUI demo (3) -> the power axis (4).** Cross-platform (1c) and the Release track (R1-R4) run alongside Phases 1-2; **v1.0 = Phases 1-2 + Release track**, and that 1.0-on-Central is the project's finish line.
